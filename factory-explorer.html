§2

It was a protocol, not a program

The single most important structural fact, and the easiest to misremember later.

There is no orchestrator. No script, no daemon, no queue runner, no state machine in code. The transcript confirms no workflow engine was ever invoked and no scheduler ran. The “state file” is a markdown document that the assistant rewrote by hand at each prompt boundary.

The factory consisted of exactly three things: a written protocol (the staged pipeline), a standing authorization granted up front in one batch, and the assistant as runtime — simultaneously the scheduler, the queue, the state store, the builder, most of the verifier, and the isolation boundary.

It worked. It also cannot run twice. The runtime was a conversation. Kill the session mid-run and the factory is a folder of markdown with nobody reading it. Every recommendation in §9 exists to fix exactly this.

Notably absent: any CI runner, any cloud agent platform, and any workflow orchestration tool. The hard gates — WebGL rendering, frame rates on a local GPU, headed-browser QA — are precisely the things a CI runner cannot do, which is why the factory was designed to run on a desk in the first place.

The default state is what actually happened: one uninterrupted session, no context compaction during the run, everything held in a single window. That was luck of scale, not design. A run of any real length hits that wall mid-build, guaranteed.

Turn the heartbeat on and kill it again. The difference is the entire argument for §9: with per-stage ledger events on disk, a fresh session reads the board, finds the in-flight lane, and resumes from the last recorded stage. The seam shows in the record; it does not show in the deliverable.

The autonomy ceiling

Plot each project by how mechanical its gate could be made, against how many repair iterations it actually took. The relationship is not subtle.

Gate objectivity vs. repair iterationsn = 7
A factory is exactly as autonomous as its gates are mechanical. The three projects with a deterministic core could be fully automated tomorrow with nothing lost. The judged ones needed rubrics, a second mind, and iteration — and the hardest still closed below full marks. Subjectivity was never eliminated. It was bounded, by converting taste into a scored rubric with a threshold. That bounding is the best available move, and it has a ceiling.

The trick that bought the most: self-supplied ground truth. One project referenced an input file that did not exist. Synthesizing that input with known properties turned “does the output resemble the source?” into “14 of 14 notes, onsets within 120 ms.” Wherever a fixture is referenced but absent, generating it buys a mechanical gate for free.

§6

Against a real factory

Benchmarked against a public reference implementation of a cloud software factory — CI-triggered agents, stages as skills, issue labels as the state machine, machine-readable review output, and a daily self-improvement workflow.

Reference source

github.com/warpdotdev-demos/cloud-factory-demo

Picked for one reason: it was available, and it is a clean, canonical example of a simple cloud factory — organized around six legible stages (Triage, Spec’ing, Implementation, Code review, Verification, Monitoring) that map onto recognizable software-development moments. That makes it a reasonable base reference for this comparison, not a claim that it is the best or only factory design out there.

The surprise: the reference factory has two mandatory human gates per task — approve the spec, then review and merge. It is a supervised assembly line. This run had zero. On the axis everyone assumes is hard, the night shift is ahead.

The gap is entirely elsewhere: the reference factory survives any particular mind being absent. That is what makes a factory a factory, and it is the one thing the night shift never had. These are different points on the same map, not a ranking — one solves supervised, code-only, indefinitely-running flow; the other solved unsupervised, mixed-media, single-burst flow on a runtime made of attention.

Distance to close: days of plumbing, not weeks of invention. The stage definitions are 80% written already, sitting in seven execution contracts. What is missing is durable state, machine-readable verdicts, and a heartbeat.

What only running taught us

Sixteen things invisible until the factory actually ran. This is the section that justifies having run it. Click any card.

The record collapsed exactly when it got interesting

The supervised precursor project produced thirteen ledger events. Under autonomous speed, the record decayed monotonically in run order — and the projects with the richest engineering stories have the thinnest record. Nine visual iterations survive as a single table inside one closure document; git shows two commits.

Ledger events per project · run orderthroughput vs. record

Documentation degrades under throughput unless the pipeline forces it, because the runtime always has something more urgent than writing history. In a factory the record is a load-bearing component — it is the input to the meta loop — not exhaust.

§9

The next factory

Target: survives session death, runs for days, keeps its meta loop on disk, and preserves everything that worked.

Substrate

Pure cloud is disqualified by this workload — the hard gates need a real GPU and a headed browser. Pure local repeats every fragility above. The answer is hybrid: local sessions become workers rather than the factory; durable queue and state live remotely as issues and labels; a scheduled heartbeat wakes a fresh worker that reads the board and resumes in-flight work from the last recorded stage. One stage of loss, maximum.

Build order

    Note the elegance of item 5: making the ledger the resume mechanism makes its collapse structurally impossible. The record stops being discipline and starts being infrastructure.

    What not to build

    No custom orchestrator daemon — the heartbeat and the board are the orchestrator. No parallel lanes in v1; serial worked, and parallelism multiplies contention while making resume combinatorial. And no attempt to mechanize taste beyond rubric, verifier, and reference board — budget a fixed number of repair passes per judged gate, then escalate. A taste gate that cannot converge is information, not a bug.
